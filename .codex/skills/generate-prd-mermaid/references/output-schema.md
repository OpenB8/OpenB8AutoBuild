# 输出 schema

按需求复杂度裁剪，但默认保持“先摘要、再正文、后图表、最后缺口”的顺序。

## 精简版

适用于单功能、单页面、单模块优化。

1. `输入缺口提醒`（如有）
2. `需求摘要`
3. `背景与目标`
4. `目标用户与核心场景`
5. `功能需求`
6. `核心流程 Mermaid 图`
7. `验收标准`
8. `待确认项`

## 完整版

适用于新产品、复杂模块、多角色或多状态流转场景。

### 第一部分：需求与立项

1. `输入缺口提醒`（如有）
   - 缺失字段
   - 影响
   - 默认处理
2. `需求摘要`
   - 用 3 到 5 句话说明问题、方案方向、收益和边界
3. `项目背景`
   - 为什么做
   - 当前问题是什么
4. `问题定义`
   - 区分真实问题与拟议方案
5. `目标用户与角色分析`
   - 主要角色
   - 决策者 / 操作者 / 管理者
6. `业务目标与成功指标`
   - 尽量量化
7. `范围定义`
   - 本期范围
   - 非本期范围
   - MVP 建议
8. `风险与依赖`
   - 数据、接口、资源、合规、跨部门依赖

### 第二部分：产品设计

1. `业务流程说明`
2. `功能模块拆解`
3. `页面结构或信息架构`
4. `状态流转说明`
5. `业务规则与约束`
6. `异常与边界场景`
7. `后台配套设计`

### 第三部分：Mermaid 图表

- `业务流程图`：默认必选
- `功能结构图`：模块较多时补充
- `状态流转图`：有审核、生命周期、工单、订单、任务状态时补充
- `时序图 / 用户旅程图`：存在多角色协作或阶段体验时补充

### 第四部分：信息缺口与待确认项

- 当前假设
- 需要业务确认的规则
- 需要技术确认的依赖
- 可能影响排期或范围的风险

## 写作要求

- 每个核心功能至少写清 `目标`、`触发条件`、`主流程`、`异常情况`
- 如果是旧系统迭代，显式标注 `新增 / 修改 / 保持不变`
- 不用“支持多种情况”“必要时处理”这类模糊表达
- 验收标准优先写成可验证的结果，而不是抽象愿景
