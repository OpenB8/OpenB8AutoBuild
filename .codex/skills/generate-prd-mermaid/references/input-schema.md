# 输入 schema

优先从用户输入中提取以下字段。字段缺失时，先提醒用户缺了什么、会影响什么，再决定是继续生成初稿还是优先追问。

## 最低输入集

- `项目名称`：没有名称时可用“暂定名称 + 业务主题”
- `项目类型`：Web / App / 小程序 / 后台系统 / AI Agent / 其他
- `业务背景`：为什么要做，当前痛点或机会点是什么
- `目标用户`：主要角色、使用者、决策者
- `核心问题`：要解决的真实问题，不要直接写方案
- `希望实现的功能`：用户期待的能力或结果

## 建议补充字段

- `成功指标`：转化率、时长、准确率、工单量、完成率等
- `上线约束`：时间、人力、预算、技术、合规、数据来源
- `已有系统`：现有页面、接口、角色、审批链、数据对象
- `范围边界`：本期不做什么，是否只做 MVP
- `参考对象`：竞品、现有流程、线下做法、法规要求

## 缺省策略

- 缺少 `成功指标` 时，先给可量化建议，并标注 `待业务确认`
- 缺少 `范围边界` 时，默认按 `MVP 优先` 处理
- 缺少 `上线约束` 时，默认按 `有限人力、短周期上线` 处理
- 缺少 `已有系统` 时，默认按 `从零设计` 处理
- 涉及审批、资金、医疗、教育、政务等高风险场景时，不擅自补齐合规口径；明确写 `需专业确认`

## 信息不足时的处理

- 先输出一个 `输入缺口提醒` 小节，明确列出缺失字段、影响范围和默认处理方式。
- `输入缺口提醒` 建议格式：
  - `缺失字段`：例如 `目标用户`、`成功指标`
  - `影响`：例如“会影响角色设计和验收标准”
  - `默认处理`：例如“先按仓库管理员/主管两类角色生成”
- 只有在信息缺口会直接改变核心流程、角色边界或合规判断时，才优先追问用户。
- 如果不影响第一版方案成型，提醒后继续输出，并在正文中单列 `假设` 与 `待确认项`。
- 不把猜测写成事实；使用 `假设`、`默认规则`、`可选方案` 标签。

## 追问与继续生成的判定

### 优先追问

- 缺少 `核心问题`，导致无法判断到底在解决什么业务问题。
- 缺少 `目标用户`，且角色差异会明显影响流程或权限设计。
- 缺少 `上线约束`，且用户又要求排期、MVP 或范围裁剪建议。
- 涉及审批、资金、医疗、教育、政务等高风险场景，缺少关键合规信息。

### 可先提醒后生成

- 缺少 `成功指标`
- 缺少 `参考对象`
- 缺少 `已有系统`
- 缺少 `范围边界`

## 输出时的最低提醒要求

- 不允许静默补全输入。
- 不允许直接假装用户已提供完整信息。
- 只要最低输入集不完整，开头必须先出现 `输入缺口提醒`。
- 即使最低输入集完整，但建议补充字段缺失明显，也应简短提醒一次。
