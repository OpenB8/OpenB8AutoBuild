# 默认交付结构

按需求复杂度裁剪，但优先保持以下顺序。

## 精简版

适用于单一功能、单页面、单模块优化。

1. `需求摘要`
2. `背景与目标`
3. `目标用户与核心场景`
4. `功能需求`
5. `核心流程 Mermaid 图`
6. `验收标准`
7. `待确认项`

## 完整版

适用于新产品、复杂模块、多角色、多状态流转的方案。

1. `需求摘要`
   - 用 3 到 5 句话说明问题、方案、收益、边界。
2. `背景与目标`
   - 为什么做
   - 解决什么问题
   - 成功指标是什么
3. `目标用户与使用场景`
   - 主要角色
   - 典型触发场景
   - 高频与高风险场景
4. `范围定义`
   - 本期范围
   - 不在本期范围
5. `用户故事`
   - 建议格式：作为 `<角色>`，我希望 `<能力>`，以便 `<价值>`
6. `功能需求`
   - 按模块拆分
   - 每项说明输入、处理规则、输出、异常情况
7. `业务规则与数据对象`
   - 关键字段
   - 状态
   - 权限
   - 约束
8. `页面或模块设计`
   - 页面入口
   - 信息层级
   - 关键交互
   - 反馈机制
9. `流程与图表`
   - 至少一个 Mermaid 图
   - 复杂需求可补充第二个或第三个图
10. `非功能要求`
   - 性能
   - 安全
   - 可用性
   - 审计或日志
11. `验收标准`
   - 主流程通过条件
   - 异常流程处理结果
   - 边界条件
12. `风险与待确认项`
   - 当前假设
   - 依赖项
   - 需要业务方确认的点

## 输出风格

- 以可评审、可继续细化为目标，不写泛泛而谈的“产品愿景文案”。
- 优先结构化表达，可用短句、表格化语义、编号列表。
- 如果用户只给一句想法，先补足合理假设，再给出完整框架。
- 若是已有系统迭代，必须标出“新增、修改、保持不变”的边界。
