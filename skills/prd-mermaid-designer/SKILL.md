# PRD Mermaid Designer

把一个产品想法、业务目标或功能请求，整理成可直接评审的需求与产品设计稿，并自动附带 Mermaid 图表。

## 何时使用

- 用户要产出 PRD、需求文档、产品方案、功能规格、页面结构、模块设计。
- 用户要把文字需求转成 Mermaid 流程图、时序图、状态图、用户旅程图。
- 用户提供的信息不完整，但希望先得到一版可评审的需求与设计初稿。

## 工作流

1. 先提炼 5 个核心输入：业务目标、目标用户、关键场景、核心动作、约束条件。
2. 如果信息不足，不要停在追问上；先明确写出 `关键假设` 与 `待确认项`，继续输出第一版方案。
3. 默认读取 `references/deliverable-template.md`，按其中的结构组织内容。
4. 默认读取 `references/mermaid-guidelines.md`，为方案生成 1 到 3 个 Mermaid 图。
5. 交叉检查正文与图表是否一致，尤其是角色、状态、模块边界、异常路径。

## 输出要求

- 默认使用中文输出，保留常见英文产品术语即可。
- 开头先给 `需求摘要`，让用户先快速判断方向对不对。
- 范围较小的需求输出精简版；范围较大的需求输出完整版。
- 每次至少给出 1 个 Mermaid 图；只要存在流程、交互或状态变化，就优先补图。
- Mermaid 图必须放在独立的代码块中，代码块语言标记为 `mermaid`。
- 图前用一句话说明图的用途，不要把大段解释写进图中。
- 如果存在不确定信息，明确标记为 `假设`、`待确认` 或 `可选方案`，不要伪装成既定事实。

## 图表选择规则

- 业务流程、页面跳转、信息架构：优先 `flowchart TD` 或 `flowchart LR`
- 用户与系统、多角色协作、接口交互：优先 `sequenceDiagram`
- 状态迁移、审批流、生命周期：优先 `stateDiagram-v2`
- 端到端体验、分阶段体验问题：优先 `journey`

## 质量门槛

- 功能模块、角色、流程步骤在正文和图表里要一一对应。
- 每个核心功能至少说明目标、触发条件、主流程、异常情况。
- 每个图都要服务于决策，不画“装饰图”。
- Mermaid 语法要稳妥，优先使用兼容性高的基础语法，不使用容易失效的高级特性。
- 如果用户给了行业、平台、业务模式或已有模块，输出必须贴合上下文，不能写成泛化模板。

## 参考文件

- `references/mermaid-guidelines.md`：Mermaid 选型、语法约束与可直接复用的图表骨架
